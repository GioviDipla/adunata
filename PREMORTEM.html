Corroborazione statica

  • cards.id nasce uuid nella migration iniziale; src/types/supabase.ts lo tipizza number
  • src/app/(app)/profile/page.tsx chiama DELETE /api/account — route non esiste
  • src/app/api/comments/[id]/route.ts cancella commenti filtrando solo per id
  • Funzioni batch SECURITY DEFINER aggiornano righe per ID senza ownership check interno
  • vercel.json schedula solo daily-sync; update-prices resta fallback manuale
  • Tutti i test browser Playwright sono test.skip()

Tesi

Se Adunata fallisce, probabilmente non sarà per assenza di feature. Fallirà per una combinazione di:

  1. schema e tipi divergenti
  2. route mutative non abbastanza difese
  3. feature real-time complesse non coperte da test end-to-end
  4. operatività fragile su cron, dati Scryfall, Supabase e provider AI
  5. grandi componenti client difficili da modificare senza regressioni

Superficie attuale: deck builder, collection, proxy PDF, goldfish, multiplayer 1v1, profili, social light, GoblinAI. La superficie di rischio è più grande della rete di test.

Risk 1 — Delete commenti senza ownership FIXED

Delete commenti senza ownership nel route handler

Impatto
Cancellazione potenziale di commenti altrui
Evidenza
src/app/api/comments/[id]/route.ts:72-75
// PRIMA (vulnerabile)
await supabase
  .from('deck_comments')
  .delete()
  .eq('id', commentId)

// DOPO (fix)
await supabase
  .from('deck_comments')
  .delete()
  .eq('id', commentId)
  .eq('user_id', user.id)
Nota
Il PATCH nella stessa route già usava .eq('user_id', user.id). L'omissione nel DELETE era accidentale. Fix applicato il 2026-05-21.

Rischi Alti High

Risk 5 — Migrazioni non autosufficienti

Migrazioni non autosufficienti rispetto ai tipi e al codice

Impatto
Fresh install, preview DB o recovery da zero possono fallire
Evidenza
I tipi includono deck_comments, deck_likes, card_likes, game_log, sync_metadata — le migrazioni locali hanno indici/alter/RPC che le usano, ma nessun CREATE TABLE.
Esempi
  • supabase/migrations/20260508050000_performance_indexes.sql crea indici su deck_comments, deck_likes, card_likes, game_log
  • supabase/migrations/20260415100000_game_log_type.sql altera game_log
  • Nessuna migration locale crea quelle tabelle
Fix raccomandato
Fare diff live → migration, o documentare esplicitamente quali tabelle sono state create fuori migration e ricostruirle. Prerequisito per affidabilità operativa.
Risk 6 — RPC SECURITY DEFINER senza ownership check

RPC SECURITY DEFINER senza ownership check interno

Impatto
Un bug applicativo può trasformarsi in scrittura cross-user

Funzioni interessate:

Aggiungere parametri p_user_id / p_deck_id e join/check interni contro owner. In alternativa spostare le funzioni in schema non esposto e restringere grant.
Risk 7 — Route pubbliche con service role per scrivere cards

Route pubbliche usano service role per scrivere cards

Decisione Finale

Prima di aggiungere nuove feature, fare una fase di hardening breve e misurabile:

  1. Chiudere i buchi P0
  2. Rendere CI verde
  3. Riallineare schema/tipi/migration
  4. Aggiungere 5–7 smoke e2e
  5. Mettere health/freshness su sync carte e AI

Dopo questi interventi, il progetto diventa molto più adatto a iterare. Senza, ogni nuova feature aumenta il costo di diagnosi dei bug esistenti.
